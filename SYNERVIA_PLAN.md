## 1. 产品定义

Synervia 是一个面向企业员工和业务团队的 AI 助理平台。它连接企业知识、业务系统与工作流程，让用户能够通过自然语言查询内部信息、分析业务问题，并在权限、审批和审计约束下执行实际操作。

一句话定位：

> 让企业员工在一个统一入口中，安全地获取知识、调用系统并完成工作。

Synervia 不是：

- 只回答文档问题的 RAG 聊天机器人；
- 只服务于 DevOps/SRE 的事故响应工具；
- 不受权限约束、可以任意执行操作的通用 Agent；
- 当前上游 `fastapi-fullstack` 生成器的简单换皮。

## 2. 核心用户与场景

### 2.1 核心用户

- 企业普通员工：查询制度、项目、客户、订单和内部流程；
- 业务负责人：汇总跨系统信息、生成报告、分派和跟踪任务；
- IT 与运营人员：配置知识库、连接器、工具、权限和审批规则；
- 安全与管理人员：审计 AI 访问的数据、执行的动作和产生的成本。

### 2.2 核心场景

1. 企业知识问答：基于制度、SOP、产品资料和历史记录生成带引用的回答。
2. 跨系统查询：组合 CRM、工单、项目管理、代码仓库和数据 API 的结果。
3. 受控任务执行：创建工单、更新记录、发送通知、生成报告或触发工作流。
4. 人工审批：高风险动作必须展示参数、影响范围并获得确认后执行。
5. 组织级治理：按用户、角色、组织和知识库隔离数据，保留完整审计记录。

## 3. 产品能力模型

```text
用户入口
  Web / API / Slack / 企业渠道
        |
身份与上下文
  用户 / 组织 / 角色 / 会话 / 权限
        |
Agent 编排层
  意图识别 / 计划 / 知识检索 / 工具选择 / 审批
        |
企业能力层
  知识库 / 连接器 / 业务 API / 工作流 / 文件 / 搜索
        |
治理与平台层
  RBAC / 数据隔离 / 审计 / 配额 / 成本 / 可观测性
```

产品能力归纳为三层：

- 懂企业：理解组织知识、用户上下文和业务数据；
- 做事情：调用工具和系统完成真实任务；
- 可治理：权限、审批、审计、隔离、成本和风险均可控制。

## 4. MVP 范围

### 4.1 MVP 必须包含

- [ ] 组织、成员、角色与工作区；
- [ ] 企业 AI 对话与会话历史；
- [ ] 多知识库管理、文档上传、同步、检索和引用；
- [ ] 至少一个只读企业连接器；
- [ ] 至少一个可执行工具；
- [ ] 高风险工具的执行前审批；
- [ ] 工具调用、审批和执行结果审计；
- [ ] 管理员配置模型、知识库、工具和成员权限；
- [ ] 基础用量、延迟、失败率与模型成本观测；
- [ ] Docker 本地环境和可重复部署流程。

### 4.2 MVP 暂不包含

- 面向公众的营销站点；
- 完整商业计费和订阅体系；
- 无审批的自主执行模式；
- 大规模多 Agent 自治协作；
- 一开始就接入大量 SaaS；
- 移动端原生应用。

## 5. 技术基线

当前仓库同时包含上游生成器源码和已生成的 Synervia 产品代码。首版技术栈已经固定，后续产品开发以 `product/synervia` 为主，不再通过反复生成覆盖产品代码。

| 层级 | 基线选择 | 原因 |
|---|---|---|
| 后端 | FastAPI + Python 3.12 | 与当前模板一致，异步和类型支持完善 |
| 前端 | Next.js 15 | 已有企业聊天、知识库和管理界面 |
| Agent | PydanticAI | 类型化依赖和工具调用更适合受控企业动作 |
| 模型 | 多 Provider，运行时选择 | 避免企业绑定单一模型供应商 |
| 数据库 | PostgreSQL + SQLAlchemy | 统一业务数据、权限、审计和会话存储 |
| 向量检索 | pgvector（MVP） | 减少基础设施数量，后续保留迁移接口 |
| 缓存/队列 | Redis + Celery | 支持异步入库、连接器同步和长任务 |
| 文档解析 | PyMuPDF，本地优先 | 降低敏感文档外发风险 |
| 文件存储 | S3/MinIO 抽象 | 兼顾本地开发与企业对象存储 |
| 身份 | 本地 JWT 起步，保留外部 IdP/JWKS | MVP 可运行，企业部署可接 SSO |
| 租户 | 多组织 + RBAC | 企业数据隔离的基础边界 |
| 可观测性 | Logfire + Prometheus + Sentry | 覆盖 Agent、接口、指标和异常 |
| 部署 | Docker Compose 起步 | 先保证开发与验证可重复 |

### 5.1 仓库与开发策略

- `product/synervia` 是 Synervia 产品的主代码和运行入口；
- `template/`、`fastapi_gen/` 和 `scripts/generate_synervia.py` 暂时保留，用于记录产品基线来源和验证生成配置；
- 日常产品功能、数据库迁移、API、前端和测试直接在 `product/synervia` 开发；
- 禁止重新生成并覆盖 `product/synervia`；生成一致性测试只能输出到临时目录；
- 产品中发现的通用问题可以选择性回补上游模板，但不要求产品与模板持续双向同步；
- 产品基线稳定后，再评估是否将 `product/synervia` 拆分为独立仓库。

当前阶段的单一事实来源：

```text
产品需求与实施状态：SYNERVIA_PLAN.md
产品运行代码：product/synervia
初始生成配置：fastapi_gen/product_profiles.py
通用生成模板：template（非产品主代码）
```

## 6. 目标主流程

### 6.1 知识问答

```text
用户提问 -> 身份与知识库权限解析 -> 查询改写 -> 混合召回
-> 重排与阈值过滤 -> 证据上下文 -> 模型回答 -> 引用展示 -> 反馈
```

### 6.2 企业动作

```text
用户提出任务 -> Agent 选择工具 -> 服务端权限校验 -> 参数校验
-> 风险分级 -> 需要时创建审批 -> 执行 -> 结果确认 -> 审计落库
```

工具不能依赖提示词完成授权。知识库访问、工具可见性和最终执行权限必须在服务端校验。

完成标准：从新检出的仓库出发，只按 README 和 Make 命令即可完成整个流程；

### Phase 1：企业知识助理

按下面顺序实施，前一批次通过验收后再进入下一批次。

#### Phase 1A：入库状态与异步任务正确性

- [ ] 修复入库服务返回错误结果、Worker 却标记成功的问题；
- [ ] 建立 `queued -> processing -> retrying -> completed / failed` 状态转换；
- [ ] 仅在重试耗尽后写入最终失败状态；
- [ ] 保存真实 vector document ID 和 chunk 数量；
- [ ] 在成功、失败和重试路径中可靠清理上传临时文件；
- [ ] 修复同步任务的成功、部分失败、取消和重试统计；
- [ ] 为上述状态转换增加自动化测试。

完成标准：数据库状态、Celery 任务状态和实际向量入库结果一致，不出现失败报成功、重试中报最终失败或临时文件泄漏。

#### Phase 1B：组织与知识库隔离

- [ ] 所有知识库操作以服务端解析的 `knowledge_base_id` 为入口；
- [ ] 禁止客户端、Agent 或任务直接指定未经授权的 collection 名称；
- [ ] 统一知识库 API、RAG API、Agent 和 Worker 的权限解析；
- [ ] 将组织、知识库和文档身份写入可验证的存储边界；
- [ ] 完善 pgvector 文档过滤、集合创建、删除和默认知识库生命周期；
- [ ] 建立两个组织之间的列表、上传、同步、检索和删除隔离测试。

完成标准：即使手工构造 API、会话或任务参数，也无法访问其他组织或其他用户无权访问的知识库。

#### Phase 1C：检索、引用与无证据策略

- [ ] 增加查询改写，并保留原始查询用于审计和评测；
- [ ] 完成向量与关键词候选召回，明确混合召回实现；
- [ ] 让 Agent 检索路径真正支持可配置重排；
- [ ] 根据实际分数语义校准有效阈值，而不是使用固定的无效默认值；
- [ ] 低于阈值或没有证据时明确拒答，不允许模型补写事实；
- [ ] 返回结构化引用，至少包含知识库、文档、chunk/page 和稳定文档 ID；
- [ ] 保证前端回答引用可以定位到原始证据。

完成标准：回答中的关键结论能够追溯到具体证据；无有效证据的问题不会生成貌似确定的企业事实。

#### Phase 1D：端到端验证与评测

- [ ] 建立文档上传、异步入库、检索、回答和引用展示的端到端测试；

## 9. 决策原则

1. 服务端授权优先于 Prompt 约束。
2. 默认只读，写操作按风险逐级开放。
3. 先做单 Agent 工具闭环，再引入多 Agent。
4. 先保证数据隔离和审计，再追求高度自治。
5. 先用少量高价值连接器验证产品，再扩展集成数量。
6. 所有异步流程必须具有明确状态、幂等键和失败恢复路径。
7. 模型生成的内容不是事实来源，关键结论必须能追溯到数据或工具结果。
